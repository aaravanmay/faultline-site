The problem

A 200 OK can still be
completely wrong.

When a tool hands your agent bad data, it usually has no idea. It doesn't crash. It doesn't warn you. It just confidently places the order, issues the refund, or reports the wrong number — and every test stays green.

That's a silent failure — the kind no eval, monitor, or unit test catches.

How it works

Silent-failure testing,
six ways.

Point faultline at your agent. It tests it six different ways — probing edge cases, fuzzing inputs, running honest scenarios, replaying recorded traces, learning rules from good runs, and stress-testing resilience — then fails the build on any silent wrong answer.

1 · Define what must hold

Feed valid-but-edge inputs and assert a behavioral rule that must always be true — regardless of model, version, or prompt change.

2 · Catch the silent wrong

If the agent acts on bad data with no error — no exception, no warning, no sign of doubt — that's a silent failure. Caught deterministically, never on a fluke.

3 · Fix it & gate it

Get the exact failure, a suggested fix, and a re-run that proves it's gone — locked into your CI so it never ships again.

Six testing modes

One tool. Six ways to
find the silent wrong.

probe, fuzz, and scenarios are the honest bug-finders — they run your real agent against real rules on honest inputs. replay and mine protect against regressions. chaos is a resilience test, clearly labeled.

bug-finder

probe

Feed valid-but-edge inputs. Assert a rule that must always hold. If the agent violates it — even once, on clean data — that's a real bug.

fl.probe(my_fn, cases, properties)
bug-finder

fuzz

Auto-generate edge inputs across the full input space. faultline finds the one combination that breaks the rule — the case you'd never write by hand.

fl.fuzz(my_fn, base_input, properties)
bug-finder

scenarios

Run your agent through honest, hard real situations with a behavioral rule. Nothing is faked — so a failure is a real agent bug, not "garbage in, garbage out."

fl.scenarios(agent, cases, invariants)
regression

replay

Re-run a recorded trace after a model, prompt, or version change. Catch silent regressions the moment they appear — before users do.

fl.replay(agent, recorded_trace)
regression

mine

Watch good runs, learn the behavioral rules yourself, then enforce them. faultline observes what "correct" looks like — and alerts when something drifts.

fl.mine(agent, good_runs)
resilience

chaos

Break a tool on purpose — wrong data, timeouts, 500s — and test if the agent has a "seatbelt." A resilience test, not a bug-finder. Honestly labeled.

faultline run faultline_suite.py

FAQ

Questions, answered.

Does faultline see my data or code? +

No. It runs entirely inside your own CI. It only injects faults into your tools and observes verdicts — your prompts, data, and code never leave your environment.

How is this different from my evals? +

Evals measure how good the answer is when everything works. faultline breaks the tools on purpose and catches the cases where your agent confidently does the wrong thing with no error. It complements evals — it doesn't replace them.

What does it work with? +

Any agent framework — LangChain, smolagents, LangGraph, CrewAI, or a raw loop. You wrap the tool functions; faultline handles the rest.

Will it trigger real side effects? +

Never. Action tools (orders, refunds, emails) are captured during a test — the real side effect never fires.

Is it really open source? +

Yes — the engine, fault library, and CLI are open source. The hosted dashboard and team features are the paid layer.
